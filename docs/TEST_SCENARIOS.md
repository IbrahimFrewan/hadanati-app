# سيناريوهات الاختبار الشاملة (E2E) — منصة حضانتي

> اختبار يدوي من البداية للنهاية عبر: تطبيق الأهل (HadanatiApp) + تطبيق الحضانة (NurseryApp) + لوحة الإدارة + الـ Backend.
> **التحضير:** مشروع Supabase مُطبَّق عليه الهجرات والدوال، مفاتيح `EXPO_PUBLIC_*` في التطبيقين، أدمن مُرقّى (`update profiles set role='admin' ...`)، وهاتفان (أو هاتف + محاكي).

**رموز:** 👨‍👩‍👧 تطبيق الأهل · 🏫 تطبيق الحضانة · 🖥️ لوحة الإدارة · ✅ النتيجة المتوقعة

## المرحلة 4 — الحجز والسعر والسعة
| # | الخطوات | ✅ |
|---|---|---|
| 4.1 | 👨‍👩‍👧 احجز شهرياً **لطفلين** وادفع | 🏫 يصل **طلبان** (طلب لكل طفل) + إشعار؛ المجموع المعروض = سعر الوحدة × 2 |
| 4.2 | 👨‍👩‍👧 احجز بالساعة 3 ساعات (qty=3) | سعر الطلب سيرفرياً = سعر الساعة × 3 (تحقق من قاعدة البيانات) |
| 4.3 | 🏫 اقبل طلباً | 👨‍👩‍👧 إشعار قبول فوري (Realtime)؛ الدفع `captured`؛ فاتورة `paid`؛ `filled` للفئة +1 |
| 4.4 | 🏫 ارفض الطلب الآخر | 👨‍👩‍👧 إشعار رفض؛ الدفع `refunded` |
| 4.5 | **سعة:** املأ الفئة بالكامل ثم اقبل طلباً إضافياً لنفس الفئة | رفض بـ «no capacity in this age group» (409) |
| 4.6 | **انتهاء:** اترك طلباً 24 ساعة (أو حرّك `expires_at`) | cron يجعله `expired` والدفع `refunded` |
| 4.7 | **سلبي:** عدّل السعر في جسم الطلب من عميل متلاعب | السعر المخزَّن سيرفري دائماً (تجاهل قيمة العميل) |

### ملاحظات
- **ترتيب التنفيذ:** المراحل بالترتيب أعلاه = دورة حياة كاملة واحدة.
- **المعروف غير المشمول:** الدفع الفعلي (stub حتى ربط البوابة)، Push للهاتف (يتطلب تسجيل التوكنات — غير مفعّل بعد)، تقويم الإغلاق/قائمة الانتظار (schema جاهز، UI لاحقاً).
